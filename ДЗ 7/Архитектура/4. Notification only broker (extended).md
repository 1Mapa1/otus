Бизнес-операции выполняются синхронно по HTTP, а NotificationService получает события о заказах и пользователях через Kafka и формирует уведомления на их основе.

## Регистрация

AuthService выступает оркестратором при регистрации: он вызывает CustomerService и BillingService напрямую.
CustomerService публикует событие о создании нового пользователя

### Diagram

```mermaid  
sequenceDiagram  
	actor C as Client
	participant AS as AuthService  
	participant CS as CustomerService
	participant BS as BillingService  
	participant K as Kafka
	participant NS as NotificationService 

	
	C->>AS: register
	AS->>CS: create account
	CS->>K: publish CustomerCreated
	CS-->>AS: code 201
	AS->>BS: create balance
	BS-->>AS: code 201
	AS-->>C: 201
	
	K-->>NS: CustomerCreated
	note over NS: save account info
```

### IDL

```IDL
register. POST /api/auth/register  
	Request:  
	{  
		login: string,
		password: string,
		name: string,
		email: string  
	}  
  
	Response:  {  }  
  
create account. POST /api/internal/customers
	Request:  
	{  
		userId: guid  
		name: string,
		email: string    
	}  
	  
	Response:  {  }  
	  
create balance. POST /api/internal/billing/accounts 
	Request:  
	{  
		userId: guid     
	}  
	  
	Response:  {  }
	
Event: CustomerCreated
	Payload:  
	{  
		userId: guid  
		name: string  
		email: string  
	}
```


## Вход

### Diagram

```mermaid

```

## Редактирование пользователя

CustomerService публикует событие о изменение пользователя

### Diagram

```mermaid  
sequenceDiagram  
	actor C as Client  
	participant CS as CustomerService  
	participant K as Kafka
	participant NS as NotificationService 

	
	C->>CS: update account
	CS->>K: publish CustomerUpdated
	CS-->>C: code 204
	
	K-->>NS: CustomerUpdated
	note over NS: update account info
```

### IDL

```IDL
update account. PUT /api/customers/me 
	Request:  
		Headers:
		{

		}
		Body:
		{  
			name: string,
			email: string,
			dateOfBirth: date | null     
		}  

	Response:  {  }  
	
Event: CustomerUpdated
	Payload:  
	{  
		userId: guid  
		name: string  
		email: string  
	}
```
## Создание заказа

OrderService вызывает напрямую BillingService и публикует событие для NotificationService 

### Diagram

```mermaid  
sequenceDiagram  
    actor C as Client
    participant OS as OrderService
    participant BS as BillingService  
    participant K as Kafka
    participant NS as NotificationService  
    
    C->>OS: create order
    note over OS: order status = Pending

    OS->>BS: withdraw
    BS-->>OS: payment result
	
    alt Paid
        note over OS: order status = Paid
        OS->>K: publish OrderPaid
        OS-->>C: 200 { status = Paid }
    else Rejected
        note over OS: order status = Rejected
        OS->>K: publish OrderRejected
        OS-->>C: 200 { status = Rejected }
    end
    
    K-->>NS: OrderPaid / OrderRejected
    note over NS: save notification
```

### IDL

```IDL
create order. POST /api/orders
	Request:  
		Headers:
		{
			Authorization: Bearer <accessToken>
		}
		Body:
		{  
			price: number  
		}  
  
	Response:  
	{  
		orderId: guid,  
		status: "Paid" | "Rejected",
		failureReason: "InsufficientFunds" | null  
	} 
	  
withdraw. POST /api/internal/billing/accounts/withdraw
	Request:  
	{  
		amount: number,
		orderId: guid,
		userId: guid
	}  
  
	Response:  
	{  
		status: "Success" | "InsufficientFunds"
	}
	
	
Event: OrderPaid  
	Payload:  
	{  
		orderId: guid,  
		userId: guid,  
		price: number  
	}
Event: OrderRejected  
	Payload:  
	{  
		orderId: guid,  
		userId: guid,  
		price: number,  
		failureReason: "InsufficientFunds"  
	}
```
